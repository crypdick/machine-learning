Implementation: Define a Performance Metric¶

It is difficult to measure the quality of a given model without quantifying its performance over training and testing. This is typically done using some type of performance metric, whether it is through calculating some type of error, the goodness of fit, or some other useful measurement. For this project, you will be calculating the coefficient of determination, R2, to quantify your model's performance. The coefficient of determination for a model is a useful statistic in regression analysis, as it often describes how "good" that model is at making predictions.

The values for R2 range from 0 to 1, which captures the percentage of squared correlation between the predicted and actual values of the target variable. A model with an R2 of 0 is no better than a model that always predicts the mean of the target variable, whereas a model with an R2 of 1 perfectly predicts the target variable. Any value between 0 and 1 indicates what percentage of the target variable, using this model, can be explained by the features. A model can be given a negative R2 as well, which indicates that the model is arbitrarily worse than one that always predicts the mean of the target variable.

Answer: To prevent overfitting your model to the specific examples in the training set. Answer: The grid search takes a list of functions you want to test in your algorithm, and iteratively tests and scores each one. How thoroughly the grid search explores the parameter space can be controlled by choosing how sparse you make the points in the grid. The grid search technique can be used if you want to brute force search all of your options for the best performing model/parameter combination. For less computationally expensive searches, we can use RandomizedSearchCV. Alternatively, if we have a very large dataset we can split it into test/train/validation sets and then validate our GridSearch results.

Answer: The K-fold cross validation is an algorithm which chunks your data into segments. Then, you reserve one of those chunks as a validation set, with the rest acting as a training set for your model. You train the model and score it. Then, you repeat the process, choosing a different chunk as the validation set. You repeat the process until each chunk has been used as a validation set exactly once. Then, you can use the summary statistics of the scores (especially mean score and variance) to benchmark the model.

This greatly increases the amount of training time, since you have to retrain the model on every conceavable combination of data chunks. If combined with grid search, it's repeating this process for every conceivable model parameters. However, you benefit from having models which have been trained and validated with all the data available to you. This is very powerful when using grid search because not only are you searching through the entire solution space, but you are getting summary statistics (especially mean and variance) for each model in the solution space. This helps find a model tailored to your needs, such as minimizing the bias-variance trade-off.

Each fold can be thought of as a trial in an experiment. More trials take more work, but they increase the confidence of the results. It prevents systematic errors occuring from not properly randomizing data before splitting it into test/training sets. It also avoids using the optimal parameters for a particular validation set, which might not be the globally optimum in the parameter space. It also lets you know how rebust your model is, since it is repeatedly being tested against a fresh validation set. The average score and score volatility lets you estimate the generalization error.
